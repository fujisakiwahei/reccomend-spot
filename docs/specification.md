# WanderMatch 仕様書

- ステータス: 初期版仕様確定
- 作成日: 2026-07-13
- 対象: MVP（観光地200件）
- 公開先: Vercel

## 1. プロダクト概要

WanderMatchは、行き先を決めずに世界の観光地を眺め、偶然の発見を楽しむための個人向けWebアプリである。

Discover画面では、世界中の観光地を1件ずつ完全ランダムに表示する。利用者はマッチングアプリのようにカードをスワイプし、気になる場所をお気に入りへ保存する。Map画面では、全観光地またはお気に入りを世界地図上で確認できる。

サーバーDB、ログイン、Cookie、アクセス解析は使用しない。観光地データはリポジトリ内のJSON、利用者ごとのお気に入りと閲覧履歴はブラウザの `localStorage` に保存する。

## 2. 目的と対象利用者

### 2.1 対象利用者

- 行き先を決めず、世界の観光地を眺めたい人
- 自分用の観光地発見ツールを求める人

### 2.2 提供価値

- 検索条件を考えなくても、世界中の観光地と偶然出会える
- 写真を中心とした短時間の操作で、気になる場所を選別できる
- 保存した場所を世界地図上で俯瞰できる
- 東京からの代表的なアクセスを見て、距離感を把握できる

### 2.3 初期版で実装しないもの

- ログイン、アカウント、複数端末間の同期
- サーバーDB、CMS、管理画面
- キーワード検索、国・地域・カテゴリ絞り込み
- レコメンド学習、嗜好分析、ランキング
- 航空券価格、便名、リアルタイム運航情報
- 営業時間、入場料金、混雑情報、天気
- お気に入り専用画面
- 共有ボタン
- PWA、オフライン利用、地図タイルの事前取得
- アクセス解析、広告、通知

## 3. 対応環境

- スマートフォンを最優先とする
- PCとタブレットにもレスポンシブ対応する
- 対象ブラウザは、実装時点でNext.jsが標準サポートするモダンブラウザとする
- 表示言語は日本語のみとする
- 観光地名、国名、都市名も日本語で表示する

## 4. 情報設計

### 4.1 ルート

| URL | 画面 | 概要 |
| --- | --- | --- |
| `/` | Discover | ランダムな観光地を1件ずつ表示する |
| `/map` | Map | 全観光地またはお気に入りを地図で表示する |
| `/spots/[slug]` | Detail | 観光地の詳細を表示する静的ページ |

### 4.2 ナビゲーション

- スマートフォンでは、DiscoverとMapを切り替える下部ナビゲーションを表示する
- PCでは、同じ2項目を画面上部に表示する
- DetailはDiscoverまたはMapから遷移する
- Detailの戻る操作はブラウザ履歴を優先し、履歴がない場合はDiscoverへ戻る
- ヘッダーメニューには「閲覧履歴をリセット」と「お気に入りをすべて削除」を置く
- リセット操作には確認ダイアログを表示する

## 5. 画面仕様

### 5.1 Discover

#### 表示内容

- 常に観光地カードを1枚、主役として表示する
- カードのおよそ70%をメイン画像に使う
- カード内に次の情報を表示する
  - 観光地名
  - 国・都市
  - カテゴリ
  - 2行までの短い紹介文
- 次の観光地のメイン画像だけを先読みし、カード切り替え時の待ち時間を抑える
- 全画像や地図タイルの一括先読みは行わない

#### ジェスチャー

| 操作 | 結果 |
| --- | --- |
| 右スワイプ | お気に入りへ保存し、閲覧済みにして次へ進む |
| 左スワイプ | スキップとして閲覧済みにし、次へ進む |
| 上スワイプ | 現在の観光地のDetailを開く |
| カードをタップ | 現在の観光地のDetailを開く |

- 上スワイプまたはタップでDetailを開いただけでは、閲覧済みにしない
- 閾値に達しなかったドラッグはキャンセルし、カードを元の位置へ戻す
- 水平方向の操作と垂直方向の操作を判定し、意図しないアクションの同時発火を防ぐ
- スワイプ中はカードを少し傾け、右方向では「保存」、左方向では「スキップ」のスタンプを表示する

#### 操作ボタン

カード下部に、安定した固定サイズのアイコンボタンを表示する。

- 戻る
- スキップ
- 詳細
- お気に入り

ボタン操作とジェスチャー操作は、同じ処理を呼び出し、結果が一致しなければならない。PCではカードのドラッグに加え、左右矢印キーでスキップとお気に入り、Enterまたは上矢印キーで詳細表示を操作できるようにする。

#### 戻る

- 取り消せるのは直前の左右スワイプまたは同等のボタン操作1件だけとする
- 直前の観光地を再表示し、その観光地を閲覧済みから外す
- 直前の操作が右スワイプだった場合、その操作によるお気に入り追加も取り消す
- 操作前からお気に入りだった観光地は、お気に入りのまま維持する
- 取り消し対象がない場合は、戻るボタンを無効にする

### 5.2 Map

- 初期表示は世界全体が見える縮尺とする
- 200件すべての観光地をピン表示する
- 近接するピンはクラスタリングし、クラスタには件数を表示する
- クラスタ選択時は、含まれるピンが判別できる縮尺まで拡大する
- 「すべて」と「お気に入り」をセグメントコントロールで切り替える
- お気に入りが0件の場合は、地図を空のままにせず空状態を表示する
- ピン選択時は、写真、観光地名、国・都市を含む概要を表示する
- 概要からDetailを開ける
- Map上の操作は閲覧済み状態を変更しない
- OpenStreetMapの著作権表記を地図上に常時表示し、UIで隠さない
- 表示中の範囲に必要なタイルだけを取得し、オフライン用ダウンロードや広範囲の先読みを行わない

### 5.3 Detail

次の順序を基本として表示する。

1. 観光地名、国・都市、カテゴリ
2. メイン画像
3. 最大3枚の追加画像
4. 短い紹介文
5. 東京からのアクセス
6. Google Mapsへの外部リンク
7. Wikipedia、公的観光サイト、公式サイトなどの情報元リンク
8. 各画像の撮影者、ライセンス、出典リンク
9. 情報の最終確認日と、交通情報が変動する旨の注記

- 画像クレジットは省略せず、画像ごとに表示する
- 外部リンクは新しいタブで開き、外部サイトであることをアクセシブルな名称で伝える
- 詳細画面からお気に入り状態を切り替えられる
- 画像取得に失敗した場合は、アプリ内の共通代替画像を表示する

## 6. ランダム選出と端末内状態

### 6.1 ランダム選出

- 初回利用時に、全200件のIDをFisher-Yates方式で均等にシャッフルしたデッキを生成する
- デッキ順を `localStorage` に保存し、再訪問しても続きから表示する
- 左右どちらの操作でも現在地を閲覧済みにし、デッキの次へ進む
- 200件を一巡するまでは同じ観光地を再表示しない
- 一巡後は新しい順序でデッキを生成し、自動的に次の周回を開始する
- お気に入り済みの観光地も、次の周回では再び表示対象になる
- 左スワイプはお気に入り解除を意味しない

### 6.2 `localStorage` のデータ

保存キーにはアプリ名とバージョンを含める。最低限、次の状態を保存する。

```ts
type PersistedWanderMatchState = {
  version: 1;
  deckSpotIds: string[];
  currentIndex: number;
  viewedSpotIds: string[];
  favoriteSpotIds: string[];
  lastAction: {
    spotId: string;
    action: "skip" | "favorite";
    wasFavoriteBefore: boolean;
  } | null;
};
```

- JSONの読み込み失敗、未知のバージョン、存在しない観光地IDを検出した場合は、安全な初期状態へ復旧する
- データ更新で観光地が追加・削除されても、存在するIDのお気に入りは可能な限り維持する
- `localStorage` が利用できない環境では、現在のタブ内だけで動作するメモリ状態へフォールバックする
- 複数タブを開いた場合は `storage` イベントでお気に入りと履歴を同期する

## 7. 観光地データ

### 7.1 件数と地域配分

| 地域 | 件数 |
| --- | ---: |
| 東・南・東南アジア | 30 |
| ヨーロッパ | 50 |
| 北米 | 30 |
| 中南米・カリブ | 30 |
| アフリカ | 20 |
| 中東・中央アジア | 20 |
| オセアニア | 20 |
| 合計 | 200 |

- 特定の国への偏りを避け、原則として1か国あたり最大5件とする
- 5件を超える場合は、その理由をリサーチ記録へ残す
- 同一施設、同一景勝地、別名による重複登録を禁止する

### 7.2 カテゴリ

1件に1個以上3個以下を設定する。

- 自然・絶景
- 歴史・遺跡
- 建築・街並み
- アート・文化
- ビーチ・リゾート
- アクティビティ
- グルメ・市場

### 7.3 ファイル構成

```text
data/
  spots/
    asia.json
    europe.json
    north-america.json
    latin-america.json
    africa.json
    middle-east-central-asia.json
    oceania.json
```

各地域ファイルは `schemaVersion` と `spots` を持つ。アプリ側では7ファイルを統合し、重複のない200件として扱う。

### 7.4 観光地のデータ型

```ts
type Region =
  | "asia"
  | "europe"
  | "north-america"
  | "latin-america"
  | "africa"
  | "middle-east-central-asia"
  | "oceania";

type Category =
  | "自然・絶景"
  | "歴史・遺跡"
  | "建築・街並み"
  | "アート・文化"
  | "ビーチ・リゾート"
  | "アクティビティ"
  | "グルメ・市場";

type ImageCredit = {
  url: string;
  alt: string;
  width: number;
  height: number;
  author: string;
  licenseName: string;
  licenseUrl: string;
  sourceUrl: string;
};

type ResearchSource = {
  title: string;
  url: string;
  type: "official" | "wikipedia" | "transport" | "other";
  checkedAt: string; // YYYY-MM-DD
};

type Spot = {
  id: string;
  slug: string;
  name: string;
  internalEnglishName?: string;
  region: Region;
  country: string;
  city: string;
  categories: Category[];
  description: string;
  coordinates: {
    latitude: number;
    longitude: number;
  };
  googleMapsUrl: string;
  accessFromTokyo: {
    departureAirport: "羽田空港" | "成田空港";
    arrivalGateway: string;
    routeSummary: string;
    estimatedDuration: string;
    transferCount: number | null;
    localTransport: string;
    note: string;
  };
  mainImage: ImageCredit;
  additionalImages: ImageCredit[];
  sources: ResearchSource[];
  lastVerifiedAt: string; // YYYY-MM-DD
};
```

### 7.5 データ制約

- `id` と `slug` は全地域で一意かつ、公開後は原則変更しない
- `slug` は小文字英数字とハイフンだけを使用する
- `description` は日本語で、カード上の2行表示とDetailの概要に共用できる長さにする
- 緯度は -90から90、経度は -180から180の範囲とする
- カテゴリは重複させない
- `additionalImages` は0枚以上3枚以下とする
- 画像はメイン画像1枚を必須とし、すべてに代替テキスト、寸法、作者、ライセンス、出典を持たせる
- 情報元は最低2件とし、基本情報用と東京からのアクセス確認用を含める
- URLは `https` を必須とし、許可したホストだけを画像表示に利用する
- `lastVerifiedAt` は、登録内容全体を最後に確認した日とする
- 営業時間、料金、航空券価格、便名は保存しない

### 7.6 東京からのアクセス表記

代表経路を、次の粒度で記載する。

> 成田空港 → シャルル・ド・ゴール空港（直行便・約14時間）→ 市内鉄道で約40分 → エッフェル塔

- 出発空港は観光地ごとに羽田空港または成田空港を選ぶ
- 直行便が一般的でない場合は、代表的な乗り継ぎ空港を1つ記載する
- 最寄りの主要空港または駅と、そこから観光地までの移動方法を含める
- 総所要時間は調査時点の目安とし、過度に精密な値を記載しない
- 「運航状況や所要時間は変動する」旨を全Detailで表示する

## 8. リサーチ仕様

### 8.1 情報源の基準

- 基本情報は、公的観光サイト、施設公式サイト、自治体、UNESCO、Wikipediaなどで確認する
- 東京からのアクセスは、空港、航空会社、鉄道会社、公的観光サイトなど別系統の情報源で確認する
- AIの事前知識だけで内容を確定しない
- 各URLが対象の主張を実際に裏付けていることを確認する
- 個人ブログや出典不明のまとめサイトは、主要な根拠にしない
- 情報が矛盾する場合は公的・公式情報を優先し、確定できない内容は記載しない

### 8.2 画像の基準

- Wikimedia Commonsを第一候補とし、必要に応じて再利用条件が明確な公式画像を検討する
- 画像ごとにファイルページを開き、作者、ライセンス名、ライセンスURL、出典URLを確認する
- Wikimedia Commons上にあるという理由だけで、個別ライセンス確認を省略しない
- クレジット必須条件、継承条件、改変表示条件を満たせない画像は採用しない
- 外部URLは元ファイルの無制限な高解像度版ではなく、画面用途に適した派生サイズを選ぶ
- 外部画像URLの直リンク可否と提供元ポリシーを確認する
- 人物が主役となる画像、商標や私有物の権利関係が不明確な画像は避ける

### 8.3 コンテキスト効率を保つ進め方

1. データ型、制約、検証スクリプトを先に確定する
2. 10件のサンプルを調査し、Discover、Map、Detailへ通す
3. データ形式と表示量を確認してから、残り190件を進める
4. 地域単位、さらに10件単位の固定バッチで調査する
5. 各バッチには、データ型、対象地域、残件数、登録済みIDだけを渡す
6. 過去バッチの長い会話や生の検索結果を次バッチへ持ち込まない
7. バッチ成果物は有効なJSONと、解決できなかった例外の短い一覧だけにする
8. 各バッチ直後に構文、必須項目、URL、重複、地域件数を自動検証する
9. 検証済みデータだけを地域JSONへ統合する
10. 地域完了時と200件完了時に、人が件数と偏りを確認する

### 8.4 リサーチ進捗

長文の調査日誌ではなく、次の最小情報だけを進捗ファイルへ記録する。

- バッチID
- 地域
- 対象観光地ID
- ステータス: `candidate` / `researching` / `validated` / `blocked`
- 最終確認日
- 未解決事項がある場合の短い理由

## 9. デザイン仕様

### 9.1 方向性

- 写真主役と情報主役を画面ごとに使い分ける
- Tinder型のカード体験と、ネオブルータリズムの強い輪郭を組み合わせる
- 太い黒線、硬いオフセットシャドウ、明確な色面を使う
- カードやボタンの角丸は控えめにし、8px以下を基本とする
- 写真自体の色を損なう強いグラデーション加工は避ける
- 装飾だけを目的としたぼかし、光の玉、グラデーション背景は使用しない

### 9.2 タイポグラフィ

- ロゴ、英字ラベル、大きな観光地名のキー書体は `Bricolage Grotesque` を第一候補とする
- キー書体は、参照画像のようなセリフ体とサンセリフ体の中間的な癖を担う
- 日本語本文、操作ラベル、小さい文字は `Noto Sans JP` を使用する
- どちらも無料かつWeb利用可能なライセンスを実装前に再確認する
- `next/font` でセルフホストし、閲覧時にGoogleへフォント取得リクエストを送らない
- キー書体が参照イメージと合わない場合は、実装前の書体比較で代替候補を提示し、合意なく変更しない

### 9.3 カラー

- ベース: 淡いシアンまたは白
- 文字と輪郭: 黒に近い高コントラスト色
- お気に入り: グリーン
- スキップ: コーラルまたは赤
- 詳細・リンク: ブルー
- 補助アクセント: イエロー
- 色だけで操作の意味を伝えず、アイコン、ラベル、形状を併用する

### 9.4 モーション

- スワイプ中は移動量に応じてカードを傾ける
- 操作確定時は画面外へ移動させ、次カードへ滑らかに切り替える
- キャンセル時は短時間で元の位置へ戻す
- レイアウト寸法は動的コンテンツでずれないよう固定・制約する
- `prefers-reduced-motion` では回転と大きな移動を抑え、短いフェードへ置き換える

## 10. 技術方針

### 10.1 基盤

- Next.js App Router
- TypeScript
- `pnpm`
- ESLint
- Tailwind CSSはNext.js標準セットアップに含めるが、デザイントークンと再利用単位を整理して運用する
- 観光地データはビルド時にローカルJSONから読み込む
- Detailは `generateStaticParams` で200件のパスを静的生成する
- API Routes、Server Actions、外部DBは使用しない
- アイコンは既存ライブラリを確認し、利用可能ならLucideを使用する

### 10.2 クライアント状態

- 状態の正本はバージョン付き `localStorage` とする
- React Contextまたは責務を限定したカスタムフックで、Discover、Map、Detailへ共有する
- 状態管理ライブラリは、React標準機能で複雑さを解消できないと確認するまで追加しない
- 永続化、マイグレーション、破損復旧を1つのモジュールへ集約する

### 10.3 地図

- 第一候補はLeaflet、React Leaflet、OpenStreetMap標準タイルとする
- React LeafletはSSR非対応のため、Map本体はクライアント専用コンポーネントとして遅延読み込みする
- クラスタリングはReact 19とReact Leafletの対応を明記するパッケージを候補とし、導入前に小さな検証を行う
- 200件表示、クラスタ展開、ピン概要、お気に入り切り替え、Vercel本番ビルドが通ることを検証ゲートとする
- タイルURLと著作権表記は1か所に集約し、提供元変更に対応できるようにする
- OSM標準タイルは可用性保証がないため、取得失敗時は再試行を繰り返さず、地図を表示できない旨と観光地概要への導線を表示する

### 10.4 画像

- `next/image` を基本とする
- `remotePatterns` で許可した画像ホストとパスだけを表示する
- JSONに任意ホストを追加しただけでは画像表示できない構成にする
- 画像寸法をJSONへ保持してレイアウトシフトを防ぐ
- Discoverのメイン画像には適切な優先度を設定し、追加画像は遅延読み込みする
- 代替画像だけはアプリ内へ同梱する

### 10.5 データ検証

ビルド前に検証コマンドを実行し、失敗時はビルドを止める。最低限、次を検証する。

- JSON構文
- データ型と必須項目
- 全体200件と地域別件数
- `id` と `slug` の一意性
- 国別件数の上限と例外記録
- カテゴリの種類、個数、重複
- 緯度・経度の範囲
- 画像枚数、寸法、クレジット
- URLの `https`、画像ホスト許可リスト
- 情報元の最低件数と確認日
- Detailルートとデータの対応

検証ライブラリを追加する場合は、実装時点の保守状況、利用実績、依存関係を確認してから決定する。

## 11. エラーと空状態

| 状態 | 振る舞い |
| --- | --- |
| 外部画像の失敗 | 共通代替画像を同じ領域に表示する |
| OSMタイルの失敗 | 地図エラーを表示し、無限再試行しない |
| お気に入り0件 | Mapのお気に入り表示に空状態を出す |
| 不正なDetail slug | Next.jsのNot Found画面を表示する |
| `localStorage` の破損 | 復旧可能なIDを残し、安全な初期状態を作る |
| `localStorage` 利用不可 | タブ内のメモリ状態で操作を継続する |
| 200件一巡 | 新しいランダム順を自動生成して継続する |
| データ検証失敗 | ビルドを失敗させ、公開しない |

## 12. アクセシビリティ

- すべてのスワイプ操作に同等のボタン操作を用意する
- アイコンだけのボタンにアクセシブルな名前とツールチップを付ける
- キーボードだけでDiscover、Map、Detailを操作できる
- フォーカス表示を明確にする
- 写真に具体的な日本語の代替テキストを設定する
- 色だけで保存、スキップ、詳細の意味を区別しない
- テキストと背景のコントラストをWCAG 2.2 AA相当で確認する
- ダイアログ表示中はフォーカスを閉じ込め、閉じた後に元の操作へ戻す
- `prefers-reduced-motion` を尊重する
- 地図だけに情報を閉じず、ピンの概要とDetailにテキスト情報を持たせる

## 13. セキュリティとプライバシー

- 認証情報、個人情報、位置情報を収集しない
- 外部URLはデータ検証時にスキームと許可条件を確認する
- 外部リンクには `noopener noreferrer` を適用する
- 画像ホストはNext.js設定で許可リスト化する
- JSON由来の文字列をHTMLとして直接挿入しない
- Content Security Policyは、画像、地図タイル、必要なスクリプトだけを許可する方針で設定する
- 利用者データは `localStorage` から外部へ送信しない
- 外部画像と地図タイルの取得時には、各提供元へ通常のHTTPリクエストが発生する

## 14. テスト方針

### 14.1 自動テスト

- データ検証テスト
- Fisher-Yatesデッキが200件を重複なく含むこと
- 一巡前に同じIDが再表示されないこと
- 一巡後にデッキが再生成されること
- 右操作、左操作、戻る、お気に入り復元の状態遷移
- 破損または旧バージョンの `localStorage` からの復旧
- 地域別JSON統合とDetail slug解決

### 14.2 Playwright

UI実装後はPlaywrightで次を確認する。

- スマートフォンで右・左・上スワイプが正しく動く
- PCでドラッグ、ボタン、キーボード操作が同じ結果になる
- 戻るが直前の1件だけを復元する
- リロード後も履歴とお気に入りが維持される
- Mapの「すべて／お気に入り」が切り替わる
- クラスタ選択、ピン概要、Detail遷移が動く
- Detailの画像失敗時に代替画像が表示される
- リセット確認ダイアログとキャンセルが動く
- 200件一巡時に自動リセットされる

スクリーンショットは `tests/` 配下へ配置し、`pnpm playwright test` で実行する。`test-results` のスクリーンショットを目視確認し、少なくともスマートフォン幅とPC幅で、文字切れ、重なり、空白崩れ、地図の欠落がないことを確認する。

Playwrightが未導入の場合は、プロジェクトルールに従い、利用者へインストールを依頼してから実行する。

## 15. パフォーマンス目標

- Discoverの初期表示では、現在カードに必要な情報と画像を優先する
- 200件の全追加画像を初期ロードしない
- Mapは遅延読み込みし、Discoverの初期JavaScriptへ地図ライブラリを含めない
- 地図表示時も、OSMタイルは現在のビューポートに必要な範囲だけ取得する
- 固定寸法とアスペクト比を指定し、画像読み込みによる大きなレイアウトシフトを防ぐ
- 実装後にLighthouseまたは同等手段でLCP、INP、CLSを計測し、重大な劣化を解消する

## 16. 実装フェーズ

### Phase 1: 基盤とデータ形式

- Next.jsプロジェクトを標準構成で作成する
- デザイントークンとフォントを設定する
- Spot型、地域JSON、検証コマンドを作成する
- `localStorage` の状態モデルを実装する

### Phase 2: 10件のリサーチと縦断実装

- 地域に偏らないサンプル10件を調査する
- Discover、Map、Detailを10件で実装する
- 地図クラスタリングとVercelビルドの互換性を検証する
- 画像クレジット、アクセス表記、カードの情報量を確認する

### Phase 3: UI検証

- スマートフォンとPCでPlaywrightを実行する
- スワイプ閾値、操作ボタン、戻る、モーションを確認する
- 参照画像と照合し、キー書体とネオブルータリズムの方向性を確認する
- データ形式の変更が必要なら、残り190件の調査前に確定する

### Phase 4: 残り190件のリサーチ

- 地域別・10件単位で調査する
- バッチごとに自動検証してから統合する
- 地域件数、国別偏り、カテゴリ偏りを確認する

### Phase 5: 統合と公開

- 200件のデータ検証を通す
- 全画面のPlaywrightと目視確認を行う
- VercelのPreviewで外部画像、OSM、静的Detailを確認する
- ライセンス表記と情報元リンクを確認する
- Productionへ公開する

## 17. 受け入れ条件

1. 有効な観光地データが地域配分どおり合計200件ある
2. Discoverで200件を重複なく一巡できる
3. 右操作でお気に入り追加、左操作でスキップ、上操作またはタップでDetail表示ができる
4. 左右どちらの操作でも閲覧済みになり、リロード後も再表示されない
5. 戻るで直前の1件だけを正しく取り消せる
6. 一巡後は新しいランダム順で自動継続する
7. Mapに全200件のピンが表示され、密集時はクラスタリングされる
8. Mapをお気に入りだけに切り替えられる
9. ピン概要から個別URLのDetailを開ける
10. 各Detailに指定情報、東京からのアクセス、情報元、画像クレジットがある
11. 画像失敗、地図失敗、空のお気に入り、不正slug、破損ストレージに対する状態がある
12. スマートフォンとPCで文字・操作・画像・地図が重ならず、主要操作がキーボードでも可能である
13. OSM著作権表記が常時見え、タイルの先読みやオフライン取得を行わない
14. Cookie、アクセス解析、ログイン、外部DBを使用していない
15. データ検証、ビルド、自動テスト、Playwrightがすべて成功する
